# i18n — Internationalization

The site uses **[next-intl](https://next-intl.dev) v3** in cookie-based mode
(no URL locale prefix). This was chosen to avoid restructuring 50+ routes
under `app/[locale]/` while still giving fully functional translations that
can be migrated incrementally.

## Supported locales

Defined in `i18n/config.ts`:

| Code | Language | Audience                          |
|------|----------|-----------------------------------|
| `en` | English  | Default / source of truth         |
| `es` | Español  | Andean / Iberian / Latin-American |
| `ja` | 日本語   | Kampo readership                  |
| `zh` | 中文     | TCM readership                    |
| `hi` | हिन्दी   | Ayurveda readership               |

## How it works

1. **`i18n/request.ts`** — resolves the active locale on every request:
   1. `rem_locale` cookie (explicit user choice)
   2. `Accept-Language` header
   3. `DEFAULT_LOCALE` ("en")
2. **`next.config.mjs`** — wraps the Next config with `createNextIntlPlugin`
   pointing at `./i18n/request.ts`.
3. **`app/layout.tsx`** — server component, wraps the tree in
   `<NextIntlClientProvider locale={…} messages={…}>`. `<html lang>` is set
   from the resolved locale.
4. **`app/api/locale/route.ts`** — POST endpoint that validates the locale
   code and writes the cookie.
5. **`components/LocaleSwitcher.tsx`** — client component in the footer;
   posts to `/api/locale` and calls `router.refresh()`.

## Using translations in a component

### Server component

```tsx
import { getTranslations } from "next-intl/server";

export default async function BrowsePage() {
  const t = await getTranslations("nav");
  return <h1>{t("browse")}</h1>;
}
```

### Client component

```tsx
"use client";
import { useTranslations } from "next-intl";

export function SearchBox({ count }: { count: number }) {
  const t = useTranslations("common");
  return <input placeholder={t("searchPlaceholder", { count })} />;
}
```

## Adding a new string

1. Add the key to `messages/en.json` under the appropriate namespace.
2. Mirror the key in every other locale file (`es`, `ja`, `zh`, `hi`).
3. Use it via `useTranslations("namespace")` or `getTranslations("namespace")`.

If a key is missing from a non-English locale, next-intl falls back to the
key name at runtime — this is a loud-enough signal in development. For
production rigor, run `npm run i18n:check` (not yet scripted; trivial to
add with a small node script diffing JSON keys).

## Phase 2 — content translation

The framework currently only covers **chrome strings** (nav, footer, common
buttons, remedy-card labels, pricing). The actual remedy content
(`site/data/extended/*.json`, root markdown files) remains English-only
because that content is pulled verbatim into the PDF and cross-referenced
with safety/references data.

Recommended path when you're ready:

1. **Structural content** (region names, taste labels, energetic terms,
   preparation types) — add namespaces to the locale JSONs, swap
   hard-coded strings at call-sites.
2. **Remedy data** — add optional `i18n: { es: { name, benefit, … } }`
   sub-objects in each extended JSON entry. The content loader already
   has a single source (`lib/content.ts`) — add a locale-aware accessor
   there.
3. **Markdown bodies** (01–08, A1, A2) — duplicate into
   `content/markdown/<locale>/` and update `resolveMarkdown` in
   `lib/content.ts` to prefer the locale folder.
4. **PDF** — extend `_build_pdf_typst.ps1` with a `-Locale` parameter
   mirroring `-Region`, pulling from the translated source tree.

Only move to URL-routed locales (`/es/encyclopedia/...`) if SEO for
translated pages becomes a requirement — at that point the
`app/[locale]/` migration becomes worth the churn.
